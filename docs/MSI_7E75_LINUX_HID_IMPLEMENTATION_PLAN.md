# MSI MS-7E75 Linux HID Implementation Plan

Status: Phase 0 report builder, Phase 1 read-only HID inventory, Phase 2 board gate, and Phase 3 dry-run reports are implemented in `src/linux/hid/`. No hardware access is enabled or implied by this plan.

## Purpose

This plan defines a staged, safety-first path for future Linux HID support for the MS-7E75 MB800 lighting path.

Current static evidence indicates the likely live path is:

```text
MS-7E75 profile
  -> EnumChipest.NUC126_MB800
  -> Class_MB_800
  -> MSI_800sLed
  -> HID_Basic
  -> MsiHid.dll
  -> HidD_SetFeature
```

Known target parameters:

- VID `0x0DB0`
- PID `0x0076`
- MI `0`
- COL `0`
- HID serial gate: first 4 serial characters parsed as hex board ID
- expected board ID: `0x7E75`

Known reports from static reverse engineering:

- `JRGB1` -> Gen1 report `0x50`, length `290`, area index `9`
- `JARGB_V2_1` -> Gen2 report `0x90`, length `302`
- `JARGB_V2_2` -> Gen2 report `0x91`, length `302`
- `JARGB_V2_3` -> Gen2 report `0x92`, length `302`
- `EZ Conn` -> Gen2 report `0x93`, length `302`

## Safety Constraints

- Documentation only.
- Do not implement HID writes in this commit or as part of this plan.
- Do not open `hidraw` devices.
- Do not run doctor.
- Do not run detect/read/write/apply.
- Do not touch `/dev/hidraw*`, `/dev/port`, SMBus, or Super I/O.
- Do not enable MS-7E75 hardware access.
- Do not claim Linux support is ready.

## Exact Non-Goals

- No device opens.
- No writes.
- No auto-detect loops that probe hardware state beyond read-only inventory metadata.
- No guesses about report descriptor semantics.
- No transport rewrite to SMBus, EC, Super I/O, or WMI.
- No release of MS-7E75 support until each gate below is separately reviewed.

## Phase 0: Static Report-Builder Only

Goal: build HID feature buffers in memory only.

Status: implemented.

Planned work:

- [x] add pure Rust structs and helpers that assemble Gen1 and Gen2 feature buffers in memory
- [x] keep report ID in byte `0`
- [x] keep lengths stable at `290` and `302`
- [x] preserve the area/port mapping from the static reverse-engineering docs
- [x] add unit tests for report length, report ID, area mapping, and port mapping
- [x] add static-layout buffer tests for simple RGB report generation

Safety gates:

- no device discovery
- no device opening
- no sysfs reads
- no `hidraw`
- no writes of any kind

Success criteria:

- [x] a buffer can be generated for each documented MS-7E75 zone entirely in memory
- [x] tests prove the byte layout matches the static documentation

## Phase 1: Detect-Only HID Inventory

Goal: list candidate HID devices without touching them.

Status: implemented.

Planned work:

- [x] enumerate candidate HID devices through read-only OS metadata only
- [x] filter strictly by VID/PID `0x0DB0/0x0076`
- [x] read only sysfs, udev, or hidapi metadata where available
- [x] record serial prefixes if exposed by metadata
- [x] print a clear unsupported message until confirmation gates are satisfied

Safety gates:

- no `SetFeature`
- no `GetFeature`
- no opens for write-capable handles
- no report transmission
- no device mutation
- no blind polling loops

Success criteria:

- [x] inventory output can identify candidate devices and state whether the target board is plausibly present
- [x] the output remains explicitly read-only and non-operational

## Phase 2: Board Gate

Goal: require board identity and explicit confirmation before any later apply path exists.

Status: implemented.

Planned work:

- [x] require DMI to match MS-7E75
- [x] require HID VID/PID `0x0DB0/0x0076`
- [x] require serial prefix parsing to `0x7E75` if serial data is available
- [x] require explicit user confirmation before any future apply command can proceed
- [x] keep the phase read-only

Safety gates:

- no writes
- no report dispatch
- no automatic retries
- no fallback to other transport families

Success criteria:

- [x] the CLI can explain why the board is accepted or rejected
- [x] all three gate types are visible in the output: DMI, HID identity, and user confirmation

## Phase 3: Dry-Run Apply

Goal: build reports and inspect them without sending them.

Status: implemented.

Planned work:

- [x] generate the same feature buffers as Phase 0
- [x] print a compact debug view of the final buffers
- [x] compare generated bytes to golden layouts from the static docs
- [x] keep the output read-only

Safety gates:

- no `SetFeature`
- no writes to a device
- no auto-detect and apply loop
- no silent fallback to another backend

Success criteria:

- [x] generated bytes match the documented buffer layouts
- [x] dry-run output is useful for review without exposing hardware access

## Phase 4: First Reviewed HID SetFeature

Goal: create a separate future commit only after review.

Planned work:

- gate the command behind an explicit flag such as `--confirm-hid-write`
- require all previous gates to pass first
- send only one simple static color report first
- avoid effect cycling, autodetect loops, and multi-report fan-out
- keep the path feature-gated and easy to disable

Safety gates:

- no writes unless every gate passes
- no bulk apply
- no fallback from failure into retries that broaden scope
- no support claim until hardware behavior is independently reviewed

Success criteria:

- a reviewer can see exactly what report will be sent
- the command fails closed if any prerequisite is missing

## Risk List

- The HID path may require additional device-state initialization that is not yet documented.
- The serial gate may be absent, truncated, or formatted differently on some systems.
- Linux `hidraw` may expose different node naming or permission behavior than expected.
- The feature report sizes may be accepted only when specific report IDs or protocol states are present.
- Future writes could have board-specific side effects if gates are too loose.

## Proposed Rust Module Layout

- `src/linux/hid/mod.rs`
- `src/linux/hid/inventory.rs`
- `src/linux/hid/gate.rs`
- `src/linux/hid/report.rs`
- `src/linux/hid/dry_run.rs`
- `src/linux/hid/cli.rs`
- `src/linux/hid/tests.rs`

Suggested responsibilities:

- `report.rs`: pure in-memory builders for Gen1 and Gen2 feature buffers
- `inventory.rs`: read-only candidate discovery and metadata formatting
- `gate.rs`: DMI, HID identity, and serial-prefix checks
- `dry_run.rs`: buffer preview and golden comparison helpers
- `cli.rs`: command wiring and confirmation handling

## Proposed CLI Shape

Suggested commands:

- `cargo run -- linux hid inventory`
- `cargo run -- linux hid gate`
- `cargo run -- linux hid build --zone JRGB1 --dry-run`
- `cargo run -- linux hid build --zone JARGB_V2_1 --dry-run`
- `cargo run -- linux hid build --zone EZConn --dry-run`
- `cargo run -- linux hid apply --zone <name> --confirm-hid-write`

Implemented now:

- `cargo run -- linux hid inventory`
- `cargo run -- linux hid gate`
- `cargo run -- linux hid dry-run --zone JARGB_V2_1 --color ff0000`

The `inventory` command reads sysfs-style HID metadata only. It does not open HID devices, does not use `SetFeature`, and does not enable MS-7E75 writes or support claims.
The `gate` command combines DMI plus read-only HID inventory metadata. It does not open HID devices, does not use `SetFeature`/`GetFeature`, and still enables no writes.
The `dry-run` command builds the Phase 0 report in memory and prints report metadata plus a hex preview. It does not open HID devices, does not use `SetFeature`/`GetFeature`, and still performs no writes.

## Test Checklist

- unit tests for each known report ID
- unit tests for each known report length
- unit tests for area/port mapping
- golden tests against the static MS-7E75 layout docs
- tests that reject non-MS-7E75 DMI strings
- tests that reject non-matching VID/PID pairs
- tests that reject missing or malformed serial prefixes
- tests that keep all Phase 1 and Phase 3 paths read-only

## Evidence Still Missing

- Linux HID inventory on a real host
- report descriptor details
- permission model for the target device node
- confirmation that the serial gate is always present and stable
- safe proof that `hidraw` is the correct future interface for this board
- any board-specific behavior that requires a non-HID backend

## Rollback / Disable Plan

- keep the Linux HID code behind a dedicated feature flag
- keep the CLI subcommands opt-in only
- keep the future write path separated from report building and inventory
- make the board gate fail closed by default
- allow disabling the entire Linux HID subtree without affecting the rest of the project

## Explicit Hardware-Access Note

This plan does not enable hardware access. It does not open devices, send HID reports, read `/dev/hidraw*`, touch `/dev/port`, or perform any SMBus or Super I/O access. It is a documentation and design plan only.
